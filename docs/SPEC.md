## 1. 개요

커플이 사진과 손글씨 낙서를 주고받으며 일상을 기록하는 **폐쇄형 소셜 앱**. "삐삐(pager)"의 아날로그 감성을 컨셉으로, 상대방의 홈 화면 위젯에 즉시 노출되는 짧은 메시지 경험을 제공한다.

앱의 중심에는 **AI 반려 펫**이 있다. 펫은 스스로 하루를 산다 — 밥을 먹고, 잠을 자고, 산책을 한다. 쓰다듬으면 지금 뭘 하는 중인지 알려주고, 하루가 끝나면 그날을 **두 사람의 그림체로 그린 일기 한 장**으로 남긴다.

### 1.1 목표

1. 사진 + 낙서를 몇 초 안에 전달하고, 홈 화면 위젯으로 즉시 확인시킨다.
2. 찌르기·사라지기 모드로 **실시간 양방향 상호작용**을 만든다. — *과제 주 옵션*
3. 펫이 하루를 살고, 말을 걸고, 그림 일기를 남긴다. — *과제 부 옵션(LLM)*
4. 누적 활동을 펫 성장과 월간 레포트로 환원해 재방문 동기를 만든다.

### 1.2 비목표

- iOS 지원 (갤럭시 단일 타깃)
- 공개 타임라인, 팔로우, 댓글 등 오픈 소셜 그래프
- 1:1 채팅 / 음성 / 영상 통화
- 현금 결제 (스토어 아이템은 코인으로만 구매)
- **유료 외부 API 사용** (예산 0원)

### 1.3 용어

| 용어 | 정의 |
|---|---|
| **그룹 (Group)** | 폐쇄형 공간의 최소 단위. 초대 코드로 가입하며 **정원 2명(커플)**. |
| **낙서 (Doodle)** | 전송 단위 1건. 사진·손그림·텍스트의 조합. |
| **일반 모드** | 다음 낙서를 보낼 때까지 유지. 레포트 집계에 반영된다. |
| **사라지기 모드** | 수신자가 확인한 시점부터 5초 뒤 삭제. 레포트에 반영되지 않는다. |
| **찌르기 (Poke)** | 내용 없이 알림만 보내는 가벼운 안부 인터랙션. |
| **펫 (Pet)** | 그룹당 1마리. 그룹 활동으로 성장한다. |
| **펫 활동 (Pet Activity)** | 펫이 지금 하고 있는 일(밥 먹기·잠·산책 등). LLM이 생성하며, 쓰다듬으면 이 활동을 말해준다. |
| **그림 일기 (Pet Diary)** | 하루치 펫 활동을 그림 한 장 + 캡션으로 남긴 기록. 모이면 일기장이 된다. |
| **그림체 모델 (Style Model)** | 일기 그림을 그릴 화풍. 처음엔 **기본 프리셋**, 낙서가 쌓이면 **사용자 그림체 LoRA**로 교체된다. |

## 2. 정보 구조 (IA) · 진입 경로

기능정의서 22–24행이 앱의 진입 경로를 규정한다.

| 진입점 | 도착 화면 | 근거 |
|---|---|---|
| 앱 실행 | **펫 키우기 (홈)** | 22행 |
| 홈 화면 위젯 탭 | **낙서 사진첩** | 23행 |
| 월간 알람 탭 | **소통** | 24행 — 7절 이슈 ② 참조 |

펫 키우기 화면이 앱의 첫 화면이자 중심이다. 낙서를 주고받는 행위가 펫을 키우고, 펫이 그 결과를 일기로 되돌려주는 순환 구조다.

## 3. 사용자 시나리오

1. A가 앱을 연다. 첫 화면에 펫이 낮잠을 자고 있다. 쓰다듬으니 *"방금 밥 먹고 졸려…"* 라고 말한다.
2. A가 사진을 찍고 그 위에 손글씨로 낙서한 뒤 *일반 모드*로 전송한다.
3. B의 홈 화면 위젯이 즉시 갱신되고 푸시 알림이 도착한다. 위젯을 탭하면 사진첩으로 들어간다.
4. B가 답장 낙서를 그려 보낸다. 펫의 경험치가 오른다.
5. 하루가 끝나면 펫이 그날의 하루를 그림 일기 한 장으로 남긴다. 처음엔 기본 그림체지만, 두 사람의 손그림이 충분히 쌓이면 **그날 이후의 일기부터 두 사람의 그림체로 그려진다.**
6. A가 *찌르기*를 보낸다. B의 화면에 즉시 알림이 뜬다.
7. 월말, 월간 레포트 알림이 도착한다. 펫 성장 폭, 이번 달 최고의 낙서, 낙서 유형 분포를 확인한다.

### 5.1 배포 제약 (README 환경표 기준)

KCLOUD VM + Cloudflare Tunnel 환경에서:

- ✅ **가능** — HTTP/REST, WebSocket, Socket.IO, SSE
- ❌ **불가** — 순수 TCP 소켓, UDP, WebRTC 미디어/DataChannel, QUIC/WebTransport
- ⚠️ gRPC는 보장 어려움

따라서 실시간 계층은 **Socket.IO(WebSocket transport)** 로 확정한다. FastAPI는 `8000`에 바인딩하고 Tunnel로 443에 노출한다. GPU 서버는 외부에 노출하지 않고 **FastAPI만 내부 HTTP로 호출**한다.

## 6. 상세 설계

### 6.1 실시간 계층 (Socket.IO) — 주 옵션

네임스페이스 `/rt`, 룸 키 `group:{group_id}`. 연결 시 토큰으로 인증하고 자신의 그룹 룸에 조인한다.

| 방향 | 이벤트 | 페이로드 | 대응 기능 |
|---|---|---|---|
| S→C | `doodle:new` | `{doodle_id, sender_id, mode, content_type, thumb_url}` | SD-5, SD-6 |
| S→C | `doodle:expired` | `{doodle_id}` | SD-6 |
| S→C | `poke` | `{from_user_id, at}` | SD-7 |
| S→C | `pet:levelup` | `{pet_id, level}` | PT-1 |
| S→C | `pet:activity` | `{pet_id, activity, utterance}` | PT-1 |
| C→S | `doodle:viewed` | `{doodle_id}` | SD-6 |
| C→S | `poke:send` | `{to_user_id}` | SD-7 |

**사라지기 모드 흐름.** 수신자가 뷰어에서 낙서를 열면 `doodle:viewed`를 보낸다. 서버는 `doodle_receipts`에 최초 확인 시각을 기록하고(중복 확인은 무시) `expires_at = now + 5s`를 세팅한 뒤, 5초 후 `deleted_at`을 채우고 미디어 파일을 실제로 지운 다음 `doodle:expired`를 룸에 브로드캐스트한다. 앱이 그 사이에 종료되어도 스케줄러의 만료 스윕이 정리한다.

**소켓과 푸시의 역할 분담.** 앱이 켜져 있으면 Socket.IO가, 백그라운드면 FCM 푸시(SD-7, SD-8, MR-1)가 알린다. 위젯 갱신은 FCM data 메시지로 트리거한다.

### 6.2 펫의 하루 — LLM 계층 (부 옵션)

**모델은 자체 GPU 서버에서 서빙한다. 외부 유료 API를 쓰지 않는다.** vLLM으로 OpenAI 호환 엔드포인트를 띄우고 FastAPI가 내부 HTTP로 호출한다.

**펫 활동 생성.** 스케줄러가 하루에 몇 차례 펫의 다음 활동을 만든다. 입력은 그룹의 최근 활동(전송 횟수, 마지막 낙서 유형, 찌르기 수)과 펫 레벨이고, 출력은 활동 종류와 그때 건네는 대사다.

```
입력: 레벨 3, 최근 24h 낙서 2건(그림 위주), 찌르기 1회
출력: { activity: "napping", utterance: "방금 밥 먹고 졸려… 그림 잘 봤어!" }
```

- 응답 형식을 강제해야 하므로 **구조화 출력(JSON 스키마)** 을 쓴다.
- `activity`는 열거값(`eating`, `sleeping`, `walking`, `playing`, `drawing`, `waiting`)으로 제한한다. 자유 문자열을 허용하면 그림 생성 프롬프트를 만들 수 없다.
- **쓰다듬기(PT-1)는 LLM을 부르지 않는다.** 현재 활동 행의 `utterance`를 그대로 돌려준다. 사용자가 연타할 수 있는 인터랙션이라 매 탭마다 추론을 돌리면 GPU가 버티지 못한다.

**그림 일기 생성.** 자정에 그날의 활동들을 모아 그림 1장 + 캡션 1문장을 만든다.

1. 그날의 `pet_activities`를 요약해 LLM이 캡션을 쓴다 — *"오늘은 낮잠을 오래 잤다. 둘이 보내준 그림이 좋았다."*
2. 활동 종류와 캡션으로 SD 프롬프트를 조립해 그림 1장을 생성한다.
3. 어떤 그림체(`style_model_id`)로 그렸는지 함께 기록한다.

### 6.3 그림체 — 기본 프리셋에서 시작해 사용자 그림체로

**기본 그림체(P1).** 그룹을 만들 때 `style_models`에 `kind='default'`, `version=0`, `status='ready'` 행을 미리 넣는다. 가입 첫날부터 일기가 그려진다.

**사용자 그림체(P2).** 그룹의 손그림 낙서(`content_type='drawing'`)가 **20장 이상** 쌓이면 학습 잡을 큐에 넣는다. GPU 서버에서 SD LoRA 파인튜닝을 돌리고 가중치 경로를 `style_models`(`kind='learned'`, `version=1`)에 기록한다. `status='ready'`가 되는 순간부터 **그 이후의 일기만** 새 그림체로 그린다. 과거 일기는 다시 그리지 않는다.

`pet_diaries.style_model_id`가 각 일기의 화풍을 기록하므로, 일기장을 넘기면 **어느 날부터 그림체가 바뀌었는지**가 그대로 보인다. 이것이 "펫이 우리 그림체를 배웠다"를 화면으로 증명하는 장치다.

> **왜 기본 그림체가 필요한가.** ① 가입 첫날은 학습할 낙서가 0장이다. ② LoRA에 쓸 만한 손그림은 현실적으로 20~30장이 필요한데 7일 안에 그만큼 쌓이지 않는다. ③ 기본 → 학습 전환이 펫 성장 서사와 정확히 맞아떨어진다.

> **리스크와 폴백.** LoRA 학습은 이 프로젝트에서 가장 먼저 무너질 지점이다. 폴백 우선순위 — ① img2img만 사용해 학습 단계 생략, ② 프리셋 몇 종만 두고 레벨에 따라 교체, ③ PT-5·PT-6b를 통째로 잘라낸다. 셋 중 무엇을 택해도 **PT-1(펫 대사)과 PT-6a(기본 그림체 일기)가 남으므로 LLM Wrapper 요건에는 영향이 없다.**

### 6.4 GPU 자원 경합

3090 한 대가 LLM 서빙과 SD를 겸한다. 24GB VRAM 안에서 둘이 부딪히므로 시간대를 가른다.

- LLM 서빙은 상주 (4-bit 양자화, 약 5~8GB)
- 일기 그림 생성은 자정 배치 1회
- **LoRA 학습은 새벽 시간대 배치**로 돌리고, 학습 중에는 LLM 활동 생성을 멈추고 마지막 활동을 유지한다

### 6.5 이번 달 최고의 낙서 (MR-3)

이미지 이해(vision)를 쓰려면 별도의 멀티모달 모델을 3090에 더 올려야 해서 규칙 기반으로 정한다.

1. 그 달의 낙서 중 **답장(`parent_id`)이 가장 많이 달린 낙서**
2. 동률이면 `stroke_data`의 획 수가 많은 낙서
3. 그래도 동률이면 최신 낙서

사라지기 모드 낙서는 삭제되므로 후보에서 제외된다.

### 6.6 인증과 "로그인 나중에 추가"

**결론: 지금 설계만 갈라두면 어렵지 않다. 나중에 붙이는 건 테이블 한 개와 라우트 두 개 분량이다.**

어려운 건 로그인 기능 자체가 아니라 **이미 쓰고 있던 익명 유저를 계정에 잇는 일(account linking)** 이다. 익명 토큰을 `users`에 직접 박아두면 "이 기기의 그 사람"과 "카카오로 로그인한 이 사람"을 같은 유저로 묶을 방법이 사라진다. 그래서 처음부터 **사람(`users`)과 인증 수단(`auth_identities`)을 분리**한다.

```
users(id, display_name)                    ← "사람". 절대 안 바뀜
auth_identities(user_id, provider, provider_uid, secret_hash)
   provider = 'device'  ← 지금. 설치 시 발급한 익명 토큰
   provider = 'kakao'   ← 나중에 행 하나 추가하면 끝
```

- **지금:** `POST /auth/register`가 `users` 1행 + `auth_identities(provider='device')` 1행을 만들고 토큰을 돌려준다. 로그인 화면 없음.
- **나중에:** 로그인된 유저가 카카오를 연동하면 **같은 `user_id`에 `provider='kakao'` 행을 하나 더 붙인다.** 기존 그룹·낙서·펫이 그대로 따라온다.
- **감수하는 것:** 소셜 로그인을 붙이기 전까지 기기 변경·앱 삭제 시 계정 복구가 불가능하다.

## 7. 미해결 사항

① **2인 정원을 어디서 강제할 것인가.** MySQL은 "자식 행 2개 이하"를 선언적으로 막지 못한다. 애플리케이션 레벨 검사 + `group_members`에 `BEFORE INSERT` 트리거를 둔다. 동시 가입 경합은 트랜잭션에서 `SELECT ... FOR UPDATE`로 막는다.

② **월간 알람의 도착 화면이 "소통"인 것이 어색하다.** 기능정의서 24행 Description이 "소통"이라 그대로 따랐다(2절). 레포트 알림이라면 레포트 뷰어(25–27행)로 가는 편이 자연스러운데, 기능정의서 우선 원칙에 따라 시트를 따랐다. **의도한 것인지 확인이 필요하다.**

③ **"Object Storage: Kcloud VM"은 성립하지 않는다.** VM은 오브젝트 스토리지가 아니다. (a) VM 파일시스템 + FastAPI 정적 서빙, (b) VM에 MinIO를 올려 S3 호환 API 사용 중 일정을 감안해 **(a)** 로 가정했다.

④ **LLM 모델 선정이 아직 열려 있다.** 3090 24GB에 4-bit 양자화로 올릴 7~8B급 한국어 모델을 골라야 한다. 후보를 벤치마크한 뒤 정한다. 캠프에서 API 크레딧을 지원받을 수 있다면 외부 LLM API로 되돌리는 편이 품질·구현 난이도 모두 유리하다 — 그 경우 vision이 생기므로 MR-3도 규칙 기반에서 되돌릴 수 있다.

⑤ **LoRA 학습 임계값 20장은 근거가 약하다.** 실제로 몇 장부터 그림체가 잡히는지 시험해 보고 조정한다.

⑥ **7일 일정 대비 기능 과다.** 36행 중 P2가 11개다. 펫 스토어·집 꾸미기·다른 그룹 탐색(EX-1~3)은 데모 임팩트가 작아 가장 먼저 잘라낼 후보다. 잘라낼 순서는 [BACKLOG.md](BACKLOG.md) C절.

⑦ **화면 설계는 별도 트랙.** Figma 화면 기획서는 다른 쪽에서 확정 후 반영하므로 이 문서는 화면 레이아웃을 규정하지 않는다. 2절의 진입 경로만 확정된 상태다.
