## C. 미룬 것 — 특히 MR-3의 vision 선정

팀 방침상 유료 외부 API를 쓰지 않는다. LLM은 자체 GPU 서버(RTX 3090)에 소형 모델을 올려 해결한다.

### C-1. 결론: MR-3의 vision 선정은 **가능하다.** 지금 안 쓰기로 했을 뿐이다

`SPEC.md` 초안은 "3090이 vision까지는 감당 못 한다"고 적었으나, 이는 **멀티모달 모델을 상주시킨다고 전제했을 때만** 참이다. MR-3는 월 1회·그룹당 1회 배치이므로 그 전제가 성립하지 않는다.

조사·검증을 거쳐 확인한 사실:

- **Qwen3-VL**은 2B / 4B / 8B / 30B / 32B / 235B로 제공되며 **Apache 2.0**(상업 사용 가능), 네이티브 VLM이다. (2025년 9~10월 공개)
- **4B AWQ의 이미지 처리 총 VRAM은 6~9GB다.** 가중치만 실측 4.43GB — AWQ가 비전 인코더·임베딩·lm_head를 고정밀로 유지하므로 3GB로 압축되지 않는다. 상주 LLM(8GB 캡)과 합치면 14~17GB로 **20GB 안에 들어가긴 하지만 헤드룸이 3~6GB뿐이다.** 큰 이미지를 넣으면 넘칠 수 있다.
- 월 1회 배치이니 여의치 않으면 그 몇 분간 vLLM을 `POST /sleep`으로 재우면 된다. 펫 대사는 `pet_activities`에 캐싱되어 있어 앱은 정상 동작한다.
- **8B는 20GB에서 상주 LLM과 공존 불가.** 쓰려면 반드시 LLM을 재워야 한다.
- 30B 이상은 20GB 단일 카드에서 구동 불가.
- 실제 저장소: `cyankiwi/Qwen3-VL-4B-Instruct-AWQ-4bit`. 32개 언어 OCR(한국어 포함).

### C-2. 도입할 때의 설계 (그대로 따를 것)

| 항목 | 내용 |
|---|---|
| 흐름 | 현행 규칙으로 **상위 5~8장을 1차 필터** → 그 소수만 VLM에 넘긴다 |
| 비교 방식 | **멀티이미지 한 프롬프트에 넣어 1장을 고르게 한다.** 장당 1~10점을 매겨 argmax 하지 마라 |
| 출력 강제 | vLLM `guided_choice`(이미지 ID 하나) 또는 `guided_json`(`{winner_id, reason}`) |
| 편향 완화 | 후보 순서를 셔플하거나 양방향 2회 후 일치 확인 |
| 배치 창 | 자정 SD 일기 생성·새벽 LoRA 학습과 겹치지 않게 (SPEC 6.4절) |
| 규칙의 지위 | 폐기하지 않는다. **1차 필터이자 배치 실패 시 폴백**으로 영구 존치 |
| 추가 구현 | 반나절~1일 |

> **왜 장당 절대점수를 매기면 안 되나.** VLM judge는 순위는 잘 매기지만 절대점수는 못 매긴다. 검증된 수치로, coarse pointwise 채점은 pairwise 비교의 약 67%에서 동률이 나고 within-prompt 상관이 r≈0.27에 그친다. 회수율은 pairwise로 바꾸면 21%에서 61%로 오른다. 후보가 5~8장뿐이니 멀티이미지 비교 한 번이면 충분하다.

> **남는 리스크.** 사진 위 낙서와 순수 손그림을 한 기준으로 비교하면 정보량이 많은 쪽으로 편향된다. 같은 `content_type` 안에서 비교하거나 프롬프트에 type을 명시해야 한다. 4B가 극도로 sparse한 낙서에서 환각할 수 있으니 `reason` 텍스트는 내부 선정용으로만 쓰고 사용자에게 바로 노출하지 않는다.

### C-3. 검토했으나 버린 대안

| 대안 | 왜 버렸나 |
|---|---|
| **CLIP aesthetic scorer** (LAION improved-aesthetic-predictor) | 장당 절대점수 argmax라 C-2의 pointwise 문제를 그대로 안는다. 학습 데이터가 사진 콘테스트·생성 이미지·로고라 손그림 도메인 시프트 우려가 있고, 무엇보다 **커플의 정성이 아니라 일반 미학**을 잰다. 또한 "SD가 이미 CLIP을 로드하니 공짜"는 **틀렸다** — SD가 로드하는 것은 CLIP **텍스트** 인코더이고, 스코어러는 CLIP **비전** 타워(약 300M)를 따로 필요로 한다 |
| **외부 API (Gemini 2.5 Flash-Lite 등)** | 비용은 문제가 아니다(후보 10장, 그룹당 월 약 2원). 문제는 둘이다. ① **커플의 사적 사진·낙서를 외부 클라우드로 보내는 것이 폐쇄형 앱 컨셉과 정면으로 충돌한다.** ② 무료 티어는 입력을 학습에 쓰고 인간 검토자가 볼 수 있다. 학습 대상에서 빠지려면 결제수단이 등록된 유료 티어여야 하는데, 그건 예산 0원 방침과 충돌한다 |

### C-4. 남은 것

| 후보 | 지금은 | 나중에 |
|---|---|---|
| **외부 LLM API (Claude 등)** | 3090에 4-bit 7~8B 한국어 모델을 vLLM으로 서빙 | 크레딧을 얻으면 품질·구현 난이도 모두 유리해진다. 엔드포인트 교체 수준 |
| **낙서 유형 자동 분류** | 필요 없음 — 전송 시 앱이 아는 값을 `content_type`에 박는다 | 그대로 둔다. vision을 써도 이득이 없다 |

### C-5. 검증되지 않은 채 남은 것

- **상주 LLM의 8GB는 `--gpu-memory-utilization 0.4`가 강제하는 상한이지 실사용량이 아니다.** 8K 컨텍스트의 KV 캐시가 그 안에서 얼마나 차는지는 재봐야 안다.
- **~~vLLM의 모델 언로드/재로드 절차를 확인하지 못했다~~ — 해결됐다.** vLLM Sleep Mode가 있다. `--enable-sleep-mode`로 띄우고 `VLLM_SERVER_DEV_MODE=1`을 주면 `POST /sleep?level=1`로 GPU를 비우고 `POST /wake_up`으로 3~6초 만에 되살린다. 프로세스를 죽일 필요가 없다. [STACK.md](STACK.md) 2절.
- 소형 VLM의 **한국어 이미지 이해 성능**은 검증 대상에서 빠졌다. MR-3는 이미지 비교이지 한글 인식이 아니라 의존도가 낮으나, 실제 낙서 몇 장으로 스팟체크를 권한다.
- **어떤 자동 지표도 커플이 실제로 아끼는 1장과 어긋날 수 있다.** 낙서에 하트·북마크 같은 명시적 선호 신호가 없기 때문이며, 이 한계는 vision을 포함한 어떤 방법으로도 해소되지 않는다. 근본 해법은 사용자가 직접 고르게 하는 것이다.

## D. 잘라낼 순서 (일정 압박 시)

7일·2인 일정에서 P2가 11개다. 데모 임팩트가 낮은 순서대로 잘라낸다.

1. **다른 그룹 펫 구경하기** (EX-1 좋아요 / EX-2 찾아가기 / EX-3 옆으로 넘기기) — 폐쇄형 서비스의 컨셉과도 다소 충돌한다
2. **펫 스토어** (PT-4) 와 **펫 집 꾸미기** (PT-3)
3. **펫 커스텀** (PT-2)
4. **그림체 학습과 전환** (PT-5, PT-6b) — LoRA 학습은 가장 먼저 무너질 지점이다. 잘라내도 **기본 그림체 일기(PT-6a)는 그대로 동작**한다
5. **배경 색상 변경** (GR-2), **설정 화면** (ST-1)

**절대 자르면 안 되는 것:** SD-6 사라지기 모드, SD-7 찌르기, SD-8 수신 푸시, RV-5 위젯 뷰어, PT-1 펫 대사, PT-6a 기본 그림체 일기. 앞의 넷이 *실시간 인터랙션*을, 뒤의 둘이 *LLM Wrapper*를 입증한다.
