## B. 충돌 해소 기록

두 문서가 서로 다른 말을 한 지점과, 기능정의서 우선 원칙에 따라 내린 결정.

| 쟁점 | 기획 문서 | 기능정의서 | 채택 |
|---|---|---|---|
| 레포트 주기 | 3절 "**주간** 레포트 (New)" | 24–27행 "**월간** 레포트" | **월간** (기능정의서) |
| 공간 단위 | "커플 전용 폐쇄형" (2인) | "그룹 만들기 / 초대 코드 / 다른 그룹 펫 구경하기" (N인처럼 읽힘) | **2인 고정.** 팀 확인 결과 커플이 메인 타깃. `groups` 명칭은 기능정의서를 따르되 정원만 2로 제한 |
| 미디어 저장 | "Object Storage: Kcloud VM" | 언급 없음 | **VM 파일시스템 + FastAPI 정적 서빙.** VM은 오브젝트 스토리지가 아니므로 기획 문서의 표현이 부정확하다 |
| AI의 정체 | "Stable Diffusion LoRA" (이미지 생성만) | 30행 "쓰다듬으면 말을 함", 31행 "AI가 그림체를 학습해서 낙서를 뱉음", 33행 "펫의 그림 일기장" | **둘 다 필요.** 30행과 33행의 캡션은 **LLM**, 31행은 **SD LoRA**. 기획 문서는 LLM을 아예 언급하지 않았으나 30행을 구현하려면 필요하다 |
| 그림 일기의 화풍 | 언급 없음 | 33행 "펫의 그림 일기장" (31행의 학습과 별개 행) | **기본 프리셋으로 시작해 학습 후 사용자 그림체로 전환.** 학습을 기다리면 첫날 일기장이 빈 화면이고, 7일 안에 LoRA 데이터가 쌓이지 않는다 |

## C. 예산 0원 제약으로 미룬 것

팀 방침상 유료 외부 API를 쓰지 않는다. LLM은 자체 GPU 서버(RTX 3090)에 소형 모델을 올려 해결한다. 그 대가로 미뤄둔 것들이다.

| 후보 | 지금은 | 예산이 생기면 |
|---|---|---|
| **외부 LLM API (Claude 등)** | 3090에 4-bit 양자화 7~8B 한국어 모델을 vLLM으로 서빙 | 품질·구현 난이도 모두 유리해진다. 코드상 엔드포인트 교체 수준 |
| **이번 달 최고의 낙서 — 이미지 이해 선정** | 규칙 기반: 답장 최다 → 획 수 최다 → 최신 (SPEC 6.5절) | vision 모델로 실제 그림을 보고 고른다. `monthly_reports.best_doodle_rule`에 규칙을 남겨뒀으므로 과거 레포트와 구분 가능 |
| **낙서 유형 자동 분류** | 필요 없음 — 전송 시 앱이 아는 값을 `content_type`에 박는다 | 그대로 둔다. vision을 써도 이득이 없다 |

캠프에서 API 크레딧을 지원받을 수 있는지 확인이 필요하다.

## D. 잘라낼 순서 (일정 압박 시)

7일·2인 일정에서 P2가 11개다. 데모 임팩트가 낮은 순서대로 잘라낸다.

1. **다른 그룹 펫 구경하기** (EX-1 좋아요 / EX-2 찾아가기 / EX-3 옆으로 넘기기) — 폐쇄형 서비스의 컨셉과도 다소 충돌한다
2. **펫 스토어** (PT-4) 와 **펫 집 꾸미기** (PT-3)
3. **펫 커스텀** (PT-2)
4. **그림체 학습과 전환** (PT-5, PT-6b) — LoRA 학습은 가장 먼저 무너질 지점이다. 잘라내도 **기본 그림체 일기(PT-6a)는 그대로 동작**한다
5. **배경 색상 변경** (GR-2), **설정 화면** (ST-1)

**절대 자르면 안 되는 것:** SD-6 사라지기 모드, SD-7 찌르기, SD-8 수신 푸시, RV-5 위젯 뷰어, PT-1 펫 대사, PT-6a 기본 그림체 일기. 앞의 넷이 *실시간 인터랙션*을, 뒤의 둘이 *LLM Wrapper*를 입증한다.
