## 0. 한 줄 요약

**지금은 "동작하는 데모"다.** 실배포·실추론·실시간까지 검증됐다(로컬 테스트 198, 소켓 E2E 7/7, `gpu:ok`).
**실사용자에게 열기 전**엔 아래 세 축이 필요하다:

1. **데이터 확보** — 시그니처 기능(그림체 학습)은 *실사용 데이터가 쌓여야* 켜진다. 즉 **연결 자체가 데이터 파이프라인의 시작**이다.
2. **검증** — 데모 시나리오가 아니라 *실제 2인 커플의 며칠치 사용*을 견디는지.
3. **실서비스 갭** — 지금은 비목표(보안·계정·백업)지만, 남에게 열려면 필요한 것.

핵심: **프론트 연결은 "끝"이 아니라 데이터 수집·검증 루프의 "시작"이다.**

## 2. 데이터 확보 (Data Acquisition)

### 2.1 ⭐ 그림체 학습 데이터 — 가장 중요, 유일하게 "쌓여야" 켜지는 기능

**무엇:** 펫 일기의 화풍을 *사용자 손그림체*로 바꾸는 SD LoRA(PT-5/PT-6b). 지금은 `learned` 요청이 **501**로 거부된다([gpu/sd_worker.py](../gpu/sd_worker.py):214). 기본 프리셋 일기(PT-6a)는 이미 동작.

**왜 데이터가 필요:** LoRA에 쓸 만한 손그림이 **그룹당 20장 이상** 필요하다([SPEC.md](SPEC.md) 6.3). 가입 첫날은 0장이고, 현실적으로 20~30장은 며칠~몇 주 실사용이 있어야 쌓인다. **→ 실사용 연결 없이는 이 기능을 만들 데이터가 아예 없다.**

**지금 할 일 (연결되는 순간부터):**
| 단계 | 내용 | 근거 |
|---|---|---|
| ① 수집 자동화 | `content_type='drawing'` 낙서를 그룹별로 집계. `stroke_data`도 함께 보존(선화 재현용) | ERD `doodles` |
| ② 임계 트리거 | 20장 도달 시 학습 잡을 큐에 넣음(현재 미구현) | SPEC 6.3 |
| ③ 학습 잡 | GPU 새벽 배치로 SD 1.5 LoRA 파인튜닝 → `style_models(kind='learned',version,status)` 기록 | STACK VRAM표(6~8GB) |
| ④ 전환 | `status='ready'` 이후의 **새 일기부터** 학습 그림체. 과거 일기는 안 다시 그림 | SPEC 6.3 |

**폴백 우선순위(SPEC 6.3, LoRA가 이 프로젝트에서 가장 먼저 무너질 지점):**
① img2img만 써서 학습 단계 생략 → ② 프리셋 몇 종을 레벨로 교체 → ③ PT-5/PT-6b 통째 컷.
셋 중 무엇이든 **PT-1(대사)·PT-6a(기본 일기)는 남아 LLM Wrapper 요건엔 영향 없음.**

> **데이터 품질 검증도 같이:** 20장이 다 낙서(스크리블)면 학습이 안 된다. "학습 가능한 손그림"의 최소 조건(획 수·캔버스 채움 비율)을 `stroke_data`로 사전 필터링할지 정해야 한다.

> **✅ 파이프라인 실측(2026-07-11).** 합성 20장으로 SD 1.5 LoRA 학습→저장→`learned` 렌더 한 사이클을 3090에서 돌려 배선을 검증했다. 결과: **peak VRAM 6.0GB**(상주 LLM ~9.8GB·SD ~3GB와 합쳐도 24GB에 듦, 또는 vLLM `sleep-mode`로 비우면 여유), 200 step **46초**(≈0.23s/step → 실학습 1~2천 step 이면 **4~8분**, 야간 배치에 적합), LoRA 가중치 **6.5MB**(그룹별 `style_models`에 저장 부담 없음), 8-bit Adam(bitsandbytes) 정상. 렌더가 학습한 화풍을 그대로 재현. **→ 3090에서 충분히 된다.** 남은 건 실데이터 20장 + 학습 잡 배선(트리거·`style_models.status` 갱신)·품질 튜닝(step 수·rank·캡션). GPU venv 에 `peft`·`bitsandbytes` 설치해 둠.

### 2.2 프롬프트·출력 품질 튜닝 데이터

**무엇:** EXAONE 펫 대사·일기 캡션의 실제 출력을 모아 프롬프트/스키마를 조정.
**왜:** guided-JSON이 스키마는 지켜도 *말투·길이·자연스러움*은 실사용 로그를 봐야 는다. 스텁 폴백이 얼마나 자주 트리거되는지도 지표.
**할 일:** 며칠간 `pet_activities.utterance`·`pet_diaries.caption` 샘플을 눈으로 리뷰 → 프롬프트([gpu.py](../backend/app/gpu.py) `pet_activity`/`diary_caption`) 조정. LLM 실패→스텁 열화 비율을 로그로 카운트.

### 2.3 기본 그림체 프리셋 검증·보강

**무엇:** 다양한 활동 조합(`sleeping+walking+eating…`)에서 SD 프리셋 일기가 일관되게 그럴듯한지.
**할 일:** 활동 조합 N개로 배치 생성해 눈으로 확인. 깨지는 조합이 있으면 프롬프트/seed 정책 조정([gpu/sd_worker.py](../gpu/sd_worker.py)).

### 2.4 (P2) 펫 스토어 아이템 자산

펫 커스텀(PT-2~4)을 켤 거면 `items` 시드 데이터 + 자산 이미지가 필요. 지금은 데모 범위 밖([BACKLOG.md](BACKLOG.md)).

## 3. 검증 (Validation) — 실사용자 붙이기 전

### 3.1 실사용 2인 시나리오 E2E

데모 스크립트가 아니라 **실제 커플 흐름**을 하루 이상 돌려본다:
온보딩(A생성→B참여) → 사진·손그림·텍스트 주고받기 → 사라지기 확인/만료 → 찌르기 → 펫 쓰다듬기(활동 로테이션) → **자정 배치로 일기 생성** → 월간 레포트 수동 생성.
체크: 두 기기에서 소켓 이벤트가 서로 실시간으로 오는가, 위젯 갱신, 별명.

### 3.2 GPU 신뢰성

| 항목 | 확인할 것 |
|---|---|
| guided-JSON | EXAONE이 `PET_ACTIVITY_SCHEMA`를 항상 지키는가, 실패 시 스텁 열화가 매끄러운가 |
| SD 실패 처리 | SD가 죽으면 그날 일기 **미생성이 정상**(빈 이미지 저장 안 함, [gpu.py](../backend/app/gpu.py) `ImageGenerationError`). 실제로 그렇게 되는지 |
| 자정 배치 | `generate_yesterday_diaries`가 실제 시각에 돌고, 활동 있는 펫만 일기가 생기는지([scheduler.py](../backend/app/scheduler.py)) |
| 재시작 회복 | mp-vllm 재시작 시 모델 재로드 ~1분, 그동안 앱은 스텁으로 버티는가 |
| GPU 공존 | vLLM(~8~9GB)+SD(~3GB) 동시 상주에서 OOM 없는지 지속 관찰 |

### 3.3 실시간 안정성

- 소켓 **재연결**: 네트워크 끊김/앱 백그라운드→포그라운드 복귀 시 자동 재연결 + 룸 재조인.
- **만료 타이머 서버 재시작 생존**: 사라지기 예약 중 uvicorn 재시작 → boot sweep이 `expires_at` 훑어 정리하는지([services.py](../backend/app/services.py) `boot_sweep`). 단일 프로세스 전제.
- 소켓 미연결 구간에서 REST 폴백으로 UX가 유지되는가.

### 3.4 부하·리소스 (2인이라 트래픽은 작지만)

- **앱 VM 4GB RAM**: MySQL(버퍼풀 128M 고정)+FastAPI+미디어 서빙이 공존. 이미지 업로드/리사이즈가 메모리 튀지 않는지([SETUP.md](SETUP.md) 4GB 튜닝).
- 이미지 업로드 크기 상한(413 `payload_too_large`)이 실제로 걸리는지.

### 3.5 데이터 정합성 (이미 하드닝됨 → 실데이터로 재확인)

동시성(그룹 정원·최초확인 멱등·레포트 동시생성)과 `DATETIME(6)` 마이그레이션은 이미 잡혀 있다. 실사용 데이터로 한 번 더 확인만.

## 4. 실서비스 갭 (지금 비목표지만 외부 공개 전 필요)

> [SPEC.md](SPEC.md) 1.2에서 **보안은 이 프로젝트의 비목표**로 선언했다. 아래는 "포트폴리오 데모"를 넘어 **실제로 남에게 열 때**의 목록이다.

### 4.1 ⭐ 미디어 접근 제어 (프라이버시)

**지금:** `/media`가 인증 없는 공개 StaticFiles([main.py](../backend/app/main.py):125). **URL만 알면 누구나 커플 사진을 본다.** 사라지기 낙서도 파일 자체는 만료 전까지 URL로 접근 가능.
**실서비스엔:** 서명 URL(만료 토큰) 또는 인증 게이트(`/media`를 `/v1` 안으로 넣고 그룹 멤버십 체크). 사라지기 낙서는 특히 URL 유출에 취약하니 우선순위 높음.

### 4.2 인증·계정 복구

**지금:** 익명 `device_uid` 토큰, 만료·갱신 없음. **재설치로 device_uid가 바뀌면 계정 상실**([API.md](API.md) 2절). `users`/`auth_identities` 분리는 돼 있어 확장 준비는 됨.
**실서비스엔:** Kakao 로그인(`provider='kakao'` 행 추가, 설계됨) + 계정 복구 경로.

### 4.3 FCM 백그라운드 푸시

**지금:** 코드 완성, `NullPushClient`로 무동작([notifications.py](../backend/app/notifications.py)). 앱이 포그라운드면 소켓으로 다 받음.
**실서비스엔:** Firebase 프로젝트 + 서비스계정 키 → `.env`에 넣으면 코드 수정 없이 켜짐. 백그라운드 알림(낙서 수신·월간 알람)이 커플 앱 UX의 핵심이라 사실상 필수.

### 4.4 백업·복구

**지금:** 없음. MySQL도 미디어도 백업 안 함.
**실서비스엔:** MySQL 일 1회 덤프(2인이라 작음) + 미디어 디렉토리 백업. 최소한 `mysqldump` cron.

### 4.5 미디어 디스크 수명주기

**지금:** 사라지기 파일은 만료 시 삭제([services.py](../backend/app/services.py) `delete_doodle_files`)되지만, **일반 낙서·일기 이미지는 무한 누적**. 100GB 디스크가 언젠가 참.
**실서비스엔:** 오래된 미디어 아카이빙 정책, 디스크 사용률 알림.

### 4.6 관측·알림

**지금:** `journalctl`만. 에러 집계·업타임 모니터 없음.
**실서비스엔:** 에러 로깅(최소 파일/외부), `/v1/health` 주기 체크 + `gpu:down`·디스크 알림.

## 5. 우선순위 로드맵

| 단계 | 목표 | 필수 항목 |
|---|---|---|
| **A. 지금 (데모)** | 프론트 연결·시연 | 이미 됨. 3.1 실사용 E2E만 한 번 |
| **B. 소프트 런치** (실제 커플 1쌍) | 우리 팀이 며칠 실사용 | 4.1 미디어 접근제어, 4.3 FCM, 3.2/3.3 GPU·소켓 안정성, 2.2 출력 품질 로그 |

## 6. 바로 착수 가능한 것 (연결 직후)

1. **낙서 손그림 카운터 대시보드** — 그룹별 `content_type='drawing'` 수를 보이게. 20장 카운트다운이 곧 LoRA 준비도.
2. **출력 샘플 로그 수집** — 펫 대사·일기 캡션을 며칠 모아 리뷰(프롬프트 튜닝 재료).
3. **미디어 접근제어 스파이크** — `/media`를 그룹 멤버십 체크 뒤로 옮기는 PoC (4.1).
4. **FCM 붙이기** — Firebase 프로젝트만 만들면 서버는 준비됨 (4.3).
